- **问题**: `_lower_match_expr` 是假实现——所有 arm 用 `MIRJump` 无条件串联，每个 arm 体都会被执行，完全没有模式匹配判定逻辑
- **实现内容**:
  - 重写 `_lower_match_expr`，采用**级联比较（if-elseif 链）**策略
  - **支持的模式类型**:

  - **环境隔离**: 每个 arm 计算前保存/恢复环境快照，避免 arm 间变量泄漏

  - **控制流结构**:
    ```
    bb_check_0: cmp(scrutinee, pat0) → if arm0_body else check_1
    bb_arm_0:   body0 → goto merge
    bb_check_1: cmp(scrutinee, pat1) → if arm1_body else check_2
    ...
    bb_merge:   phi(arm0_result, arm1_result, ...)
    ```

### 三、问题总结与根因分析

#### 问题 1: MIR match 降级完全错误 🔴 最高
**严重程度**: 🔴 最高
**表现**: `_lower_match_expr` 中所有 arm 用 MIRJump 无条件串联，每个 arm 体都会被执行，没有模式匹配判定
**根因**: MIR 降级器是在原型基础上快速迭代出来的，match 降级一直是占位实现，之后没人回头补全
**影响**:
- 所有含 match 的程序在 MIR/LIR 层语义错误
- ADT/模式匹配功能形同虚设
- 依赖 MIR 的后端（C LIR 路径、Wasm、Cranelift、Native）都受影响

#### 问题 2: MIR 循环变量未绑定 🟠 高
**严重程度**: 🟠 高
**表现**: for 循环变量从未加入 `self.env`，循环体内引用返回 None，生成空 SSA 名；列表推导式循环变量绑定到可迭代对象本身而非当前元素
**根因**: MIR 降级的循环实现不完整，只搭了 CFG 框架，变量绑定遗漏
**影响**:
- 所有含 for 循环的程序在 MIR/LIR 层行为错误
- 列表推导式在 MIR 层结果不正确
- 影响范围广（循环是最常用的语言特性之一）

#### 问题 3: SSA 构建不严格 🟠 高
**严重程度**: 🟠 高
**表现**: 赋值直接覆盖 `self.env` 中的 SSA 名，循环头无 Phi 节点，MIRStore 与 SSA 语义冲突
**根因**: MIR 降级器的 SSA 是简化版本，没有实现完整的 Cytron 算法和支配边界计算
**影响**:
- 所有依赖 SSA 的优化（CSE、LICM）建立在不稳固的基础上
- 变量重名可能导致值被意外覆盖
- 无法引入更高级的 SSA 优化

#### 问题 4: sys.path hack 持续增加 🟠 高
**严重程度**: 🟠 高
**表现**: 从 17 处增加到 19 处，新代码（lir_c_backend.py）也沿用了坏模式
**根因**: 包结构不标准，没有建立编码规范，新人/新代码自然沿用现有模式
**影响**:
- 导入顺序问题、调试困难
- 包工具（pip install -e）无法正常使用
- 技术债具有传染性，越拖越难修

#### 问题 5: Pass 管理器重复代码多 🟡 中
**严重程度**: 🟡 中
**表现**: 6 个方法重复实现 HIR 表达式的递归遍历，约 500 行重复代码
**根因**: 没有引入 Visitor 模式，每个 Pass 各自实现遍历逻辑
**影响**:
- 维护成本高，新增 HIR 节点类型需要修改 6 个地方
- 容易遗漏，不同 Pass 的遍历行为可能不一致
- bug 修复需要在多处同步

### 三、问题总结与根因分析

#### 问题 1: C 后端绕过 IR 管线 — "两个编译器"问题
**严重程度**: 🔴 最高
**表现**: `compiler_pipeline.py` 中 C 后端直接接收 AST 生成代码，完全不走 HIR/MIR/LIR 和优化 Pass
**根因**: C 后端开发早于三层 IR 设计，之后一直没做迁移
**影响**:
- IR 层的所有优化成果无法惠及最成熟的后端
- 测试和验证路径分裂，IR 改动只能靠 IR 层测试验证
- 优化 Pass 的实际效果无法通过编译运行来衡量

#### 问题 2: 跨层命名不一致 — 持续的认知税
**严重程度**: 🟠 高
**表现**: 函数调用字段名在四层中换了三次（callee→function→callee→func_name）
**根因**: 各层独立开发，没有统一的命名规范
**影响**:
- 已导致 DCE 和 Inlining 的字段名 bug
- 每次写跨层代码都要回忆"这层叫什么"
- 增加 bug 风险和维护成本

#### 问题 3: sys.path hack — 最普遍的技术债
**严重程度**: 🟠 高
**表现**: 19 处 `sys.path.insert`，核心代码 12 处，测试 7 处
**根因**: 包结构不标准，`ir/` 目录内模块直接导入不带包前缀
**影响**:
- 导入顺序问题
- 调试困难
- 包工具（pip install -e）无法正常使用
- 技术债传染到测试代码

#### 问题 4: MIR SSA 构建正确性存疑
**严重程度**: 🟠 高
**表现**: 循环变量不生成 SSA 新版本、赋值直接覆盖 env、Phi 只取第一个 source
**根因**: MIR 降级器的 SSA 构建是简化版本，没有实现完整的支配边界算法
**影响**:
- 所有依赖 SSA 的优化（CSE、LICM）都建立在不稳固的基础上
- Phi 节点语义不正确，后端代码生成可能出错
- 变量重名可能导致值被意外覆盖

#### 问题 5: Pass 管理器静默错误吞噬
**严重程度**: 🟡 中
**表现**: 3 处 `except Exception: pass`，优化失败完全无感知
**根因**: 为了不影响主流程而简单吞掉错误
**影响**:
- 用户不知道优化是否生效
- 调试困难，失败了也没有日志
- 优化 Pass 的实际覆盖率可能远低于预期

### 四、下阶段方向与理由

#### 总体策略: 先打地基，再盖高楼
接下来 3 轮（第 4-6 轮）从"功能开发"转向"架构治理 + 基础夯实"。

**核心理念**: 
1. 先打通端到端管线，让 IR 的价值可验证
2. 再夯实 SSA 基础，让优化有正确的前提
3. 最后治理技术债，为后续加速铺路

**聚焦任务**: C 后端接入统一 IR 管线

1. 这是当前最大的架构裂痕，不解决的话 IR 层的工作始终是"自嗨"
2. 打通后，所有优化 Pass 的效果都能通过 C 后端编译运行来验证
3. 可以统一测试路径，减少维护成本

**预期成果**:

- 现有 C 后端测试全部通过
- 优化 Pass 效果可通过 C 后端实测

#### 第 5 轮: 基础夯实 — 修复 MIR SSA + 实现 CSE
**聚焦任务**: 修复 MIR SSA 构建正确性 + 实现公共子表达式消除

**为什么第二个做**:
1. SSA 是所有经典优化的基础，不修好的话优化都是空中楼阁

**预期成果**:
- MIR 层 SSA 构建正确（循环变量、if-else 汇合、赋值不破坏单赋值）

- 新增 SSA 正确性测试

**为什么第三个做**:
1. 这两项都是"全项目级"的重构，放在一起做减少震荡
2. 前两轮是功能建设，这一轮是地基加固
3. 越早做成本越低，代码越多重构代价越大
4. 为后续更多模块开发扫清障碍

**预期成果**:
- 所有导入使用标准包路径
- 消除全部 sys.path hack

- 包可通过 `pip install -e` 正常安装
